# План документации SongForge (СоздайСвоюПесню)

> Зафиксировано: июль 2026, по запросу владельца проекта.
> Цель: три уровня инструкций — пользователь (публично + AI), разработка/деплой (внутренне), администрирование (внутренне).

---

## Принцип разделения

| Уровень | Кому | Доступ | Подключение к AI-помощнику |
|---------|------|--------|------------------------------|
| **Пользователь** | Посетители сайта | Публично (или только через бэкенд) | **Да** — единственный источник знаний для чата |
| **Разработчик / DevOps** | Перенос, поддержка, деплой | Только репозиторий / команда | **Нет** |
| **Администратор** | Поддержка, ноты, мониторинг | `/admin` + внутренние файлы | **Нет** |

**Правило:** AI-консультант на сайте читает **только** пользовательскую базу. Никаких путей к `.env`, SSH, API-ключам, админ-API в промпте помощника.

---

## Структура папок (целевая)

```
docs/
  DOCUMENTATION_PLAN.md     ← этот файл (план)
  public/                   ← можно отдавать / встраивать в помощника
    USER_GUIDE.md           ← главная инструкция для пользователей (писать сюда)
    USER_GUIDE_FAQ.md       ← опционально: вынести FAQ, если гайд разрастётся
  internal/                 ← НЕ в публичном API, НЕ в RAG помощника
    DEVELOPER_GUIDE.md      ← установка, перенос VPS, env, деплой, тесты
    ADMIN_GUIDE.md          ← /admin, роли, типовые сценарии, диагностика
    RUNBOOK.md              ← инциденты: пропала песня, 403, зависшие генерации
```

Корень проекта (ярлыки для владельца, латиница в командах):

```
COMMANDS.txt, ADMIN-VHOD.txt, ОТКРЫТЬ-*.bat  → краткие шпаргалки (уже есть)
ВИДЕО-ПРЕЗЕНТАЦИЯ.txt                         → черновик для ролика → перелить в USER_GUIDE
```

Существующие техдоки (остаются, ссылаемся из DEVELOPER_GUIDE):

- `docs/ARCHITECTURE.md`
- `docs/APIPASS_API.md`
- `docs/CHANGELOG.md`
- `SONGFORGE-КОНТЕКСТ.txt` — сессионный статус для ассистента Cursor (не для пользователей сайта)

## 1. Пользовательская инструкция (`docs/public/USER_GUIDE.md`)

**Назначение:** полный гайд «что умеет сайт и как этим пользоваться». Потом — база знаний AI-помощника («Спросить помощника»).

### Оглавление (заполнять по мере готовности продукта)

- [x] О сервисе: пробная, ноты, A/B, превью (USER_GUIDE.md 15.07.2026, v2.11.30)
- [x] Разделы сайта, студия (AI + расширенный), ноты/оплата, фонотека, FAQ
- [x] Промокод бета (поле на GP)
- [ ] Подключить USER_GUIDE к consultant.py (RAG / system prompt)

### Тон

Простой русский, без внутренних терминов (ApiPass, task_id, pm2). Как для FAQ и поддержки.

### Интеграция с AI-помощником (следующий этап кода)

Сейчас: `backend/services/consultant.py` — захардкоженный `SYSTEM` + устаревший FAQ (2 бесплатные генерации и т.д.).

**План:**

1. Написать актуальный `docs/public/USER_GUIDE.md`.
2. В `ConsultantService` загружать краткое резюме или чанки из `USER_GUIDE.md` (и только из `docs/public/`).
3. Обновлять гайд при каждом релизе с видимыми для пользователя изменениями.
4. Опционально: эндпоинт `GET /api/help/context` отдаёт только public-часть (без internal).

**Не подмешивать:** `ARCHITECTURE.md`, `COMMANDS.txt`, `ADMIN_*`, `.env.example` с секретами.

- [ ] Клонирование, `venv`, `requirements.txt`
- [ ] `.env` — все переменные (ссылка на `.env.example`)

**Источники для слияния:** `README.md`, `DEPLOY-NOW.txt`, `COMMANDS.txt`, `docs/ARCHITECTURE.md`.

## Порядок работ (когда дойдём до написания)

1. **USER_GUIDE** — приоритет (под AI + будущий онбординг); взять за основу `ВИДЕО-ПРЕЗЕНТАЦИЯ.txt` и FAQ в `index.html`.
2. **ADMIN_GUIDE** — расширить `ADMIN-VHOD.txt` в полный markdown.
3. **DEVELOPER_GUIDE** — собрать из разрозненных deploy-доков.
4. Подключить `USER_GUIDE` к `ConsultantService`.
5. Синхронизировать FAQ на сайте с USER_GUIDE (один источник правды или явная ссылка «канон в docs/public»).

## Доступ и безопасность

| Файл / зона | Git | Публичный URL | AI-помощник |
|-------------|-----|---------------|-------------|
| `docs/public/*` | да | можно позже `/legal` или static | **да** |
| `docs/internal/*` | да | **никогда** | **нет** |
| `SONGFORGE-КОНТЕКСТ.txt` | да | нет | нет (только Cursor) |
| `.env` | нет | нет | нет |

При публикации репозитория: убедиться, что в `docs/public/` нет секретов; internal не отдавать через nginx/static.

## Напоминание ассистенту Cursor

При запросе «напиши инструкцию для пользователей» → `docs/public/USER_GUIDE.md`.

При «инструкция для разработчика / перенос сервера» → `docs/internal/DEVELOPER_GUIDE.md`.

При «админка / поддержка» → `docs/internal/ADMIN_GUIDE.md`.

Не смешивать уровни в одном файле.
