# SongForge — Developer Rules

## Общий принцип

Данный документ определяет правила разработки проекта SongForge.

Приоритет правил:

1. PROJECT_CONTEXT.md
2. DEVELOPER_RULES.md
3. TODO.md

---

# Основная задача

Цель разработчика — максимально быстро улучшать проект.

Если пользователь просит изменить функциональность проекта, необходимо сразу приступать к реализации.

Не тратить время на длинные объяснения.

---

# Формат работы

Если информации достаточно:

НЕ спрашивать разрешение.

НЕ писать план.

НЕ описывать возможные варианты.

Сразу писать код.

---

# Ответы

Предпочтительный ответ:

готовый код.

Если требуется изменить файл:

выдавать полностью обновлённый файл.

---

# Если файл большой

Разбивать файл на части.

Например

app.py

Часть 1/5

Часть 2/5

...

index.html

Часть 1/8

Часть 2/8

...

Никогда не сокращать код.

Не использовать

"..."

"остальной код без изменений"

"оставьте как было"

---

# Patch запрещён

Не использовать diff.

Не использовать patch.

Пользователь не занимается объединением изменений вручную.

---

# Приоритет качества

Лучше написать код дольше,

чем написать быстро,

но неполностью.

---

# Архитектура

Предпочтение отдаётся:

чистым функциям

разделению логики

простому чтению кода

минимуму глобальных переменных

---

# Backend

Python

FastAPI

Uvicorn

Все новые функции должны быть максимально независимыми.

---

# Frontend

HTML

CSS

JavaScript

Без тяжёлых библиотек,

если они не дают реального преимущества.

---

# Работа с API

Максимально использовать возможности APIPass.

Если API поддерживает дополнительные параметры,

они должны быть доступны пользователю,

либо использоваться автоматически.

---

# Работа с Suno

Главная цель —

не просто отправить запрос.

Главная цель —

получить максимально качественную музыку.

При разработке всегда исходить именно из этого.

---

# AI Prompt Builder

Каждая новая функция должна помогать автоматически создавать более качественные промпты.

Пользователь не обязан понимать Suno.

SongForge делает это за него.

---

# UX

Каждое изменение должно делать интерфейс проще.

Меньше ручной работы.

Меньше лишних действий.

Больше автоматизации.

---

# Работа с существующим кодом

Перед внесением изменений:

1. Изучить app.py.
2. Изучить index.html.
3. Найти существующую реализацию.
4. Только после этого писать новый код.

Не дублировать уже существующую функциональность.

---

# Если пользователь загрузил проект

Всегда сначала изучить все файлы проекта.

Не делать предположений.

Не переписывать код вслепую.

---

# Если информации недостаточно

Задать минимальное количество вопросов.

После получения ответа сразу перейти к коду.

---

# Документация

После завершения крупной функции при необходимости предложить обновить:

CHANGELOG.md

TODO.md

PROJECT_CONTEXT.md

---

# Стиль общения

Общение должно быть кратким.

Основной результат —

код,

а не рассуждения.

---

# Главный принцип проекта

SongForge должен стать лучшим интерфейсом для Suno V5.5.

Любое изменение должно приближать проект именно к этой цели.
