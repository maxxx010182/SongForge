## Экран генерации — факты и прогресс

Блок «Создаём твой лучший трек…» + интересные факты внизу (`#funFact`, `#generatingMsg`).

### Проблема 1: мало фактов, одни и те же по кругу

**Сейчас (v2.1.0):** в `index.html` массив `funFacts` — всего **8 фраз**, случайный выбор каждые 7 сек. Быстро повторяются.

**Нужно:**
- [ ] Расширить библиотеку до **30–50+** фактов (музыка, AI, Suno, жанры, история песен)
- [ ] Не показывать один и тот же факт дважды подряд
- [ ] Опционально: факты по контексту (жанр/настроение выбранного трека)
- [ ] Ротация статусов `genMsgs` — тоже обогатить, сейчас 6 шагов с дублями

### Проблема 2: «Готово!» слишком рано

**Сейчас:** при ~**85–95%** прогресса UI уже пишет «Готово!» / «Секунды до результата…», хотя Suno ещё генерирует **1–3 минуты**.

**Причина в коде (`index.html`):**
- Прогресс искусственно капится на **95%** (`elapsed/240000*95`), пока идёт polling
- Сообщения переключаются по формуле `floor(pct/17)` — последний шаг `genMsgs[5]` = «Готово!» наступает при pct ≥ 85
- Реальный ответ API приходит позже (`pollMusicStatus`, до 48×5 сек)

**Нужно:**
- [x] «Готово!» и **100%** — только при `hideGenerating({success:true})` после success API *(v2.1.5)*
- [x] До успеха — кап прогресса 88%, без «Готово!» в таймере *(v2.1.5)*
- [x] Polling 3 сек вместо 5 *(v2.1.5)*
- [x] Факты 30+ без повтора подряд *(v2.1.5)*
- [ ] Полная привязка % к `progress_hint` с бэкенда

**Файлы:** `index.html` — `funFacts`, `genMsgs`, `showGenerating`, `setGeneratingStatus`, `pollMusicStatus`, `hideGenerating`.

## История — полный трек без оплаты (задел на будущее)

**Симптом (замечено июль 2026):** во вкладке **Фонотека → История** кнопки ▶ A / ▶ B дают слушать **весь трек**, хотя в UI написано «Превью 30 сек • не куплено».

**Почему так:**
- В модалке результата (`toggleVariantPreview`) есть обрезка на **30 сек** (`currentTime >= 30` → pause).
- В истории (`renderHistory` → `hist-preview-btn`) вызывается `playAudio(url)` **без лимита** — играет полный `music_url_a/b` с CDN Suno/APIPass.
- API `GET /api/history` отдаёт клиенту **полные URL** — даже с лимитом на фронте ссылку можно скопировать из Network и слушать/скачать напрямую.

**Что сделать (по приоритету):**

### Быстрый фикс (только UI) — сделано v2.1.5
- [x] История: `playDemoAudio` — лимит 30 сек (как в модалке)
- [x] Плеер внизу — превью обрезается на 30 сек

### Надёжный фикс (бэкенд) — частично v2.1.5
- [x] `/api/history` — без полных URL, только `has_preview_a/b`
- [x] `GET /api/history/{id}/preview?variant=a|b` — URL по запросу с проверкой сессии
- [ ] Прокси/ffmpeg trim — полные URL всё ещё в preview endpoint (следующий этап)
- [ ] Полные URL — только после покупки (`/api/library`, purchase response)
- [ ] Опционально: signed URL с коротким TTL для превью

### Данные
- Сейчас в `generations` хранятся только полные `music_url_a/b` (до ~4 мин)
- В `showcase_tracks` уже заложены `preview_url_a/b` — можно переиспользовать идею для истории

**Файлы:** `index.html` — `renderHistory`, `playAudio`, `toggleVariantPreview`; `cabinet_service.py` — `list_history`; `app.py` — новый preview endpoint.

**Пока не менять** — зафиксировано для следующей правки монетизации.

## Исправлено: AI-продюсер не генерировал стихи (июль 2026)

**Симптом:** вместо поэтического текста Suno пел описание пользователя дословно.

**Причина:** `AiPromptComposer` просил ЯндексGPT вернуть title + lyrics + style в одном JSON — модель часто копировала идею; при ошибке срабатывал fallback с вставкой идеи в куплет.

**Исправление:**
- Текст песни теперь генерируется **отдельным** вызовом `generate_lyrics()` с промптом «поэт-песенник»
- Composer отвечает только за style/title/параметры
- Проверка `lyrics_look_lazy()` + повторная генерация при «ленивом» ответе
- Fallback без дословной вставки описания

**Файлы:** `prompt_builder.py`, `ai_prompt_composer.py`, `utils/text.py`
