# Pre-Beta Launch Checklist (Критично для беты 5–20 человек)

**Дата создания:** 14 июля 2026
**Цель:** Минимальный набор доработок, чтобы запустить бету без крупных рисков (abuse, падения, потеря денег).
**Принцип:** Только must-have. Всё остальное — после беты или параллельно.

## Статус на сейчас
- Закрыто ранее: webhook security, CORS, оплата GetPlatinum (с возвратом текста и нот).
- В процессе: базовый rate limiting (фронтенд cooldown на текст и создание песни).

## Приоритет 1 — Критично перед бетой (делать в первую очередь)

1. **Rate limiting (продолжить)**
   - Текущий фронтенд (10 сек на создание песни, лимит 4 на текст) — хорошо для начала.
   - Рекомендация: добавить серверный rate limit (slowapi или простой middleware) хотя бы на:
     - /api/create-song
     - /api/auth/email/request
     - /api/produce + /api/generate-lyrics
   - Лимиты: 5–8 запросов в минуту на IP/guest, мягче для авторизованных.
   - Зачем: защита от ботов, которые обходят фронтенд.

2. **Защита пробной генерации (guest trial)**
   - Сейчас только по cookie → легко обойти (новый браузер, инкогнито, несколько устройств).
   - Нужно: привязка к fingerprint + IP + ограничение по времени/количеству.
   - Или хотя бы серверная проверка + лимит 1 пробная на комбинацию признаков.

3. **Лимит параллельных генераций**
   - Нет ограничения → worker может задохнуться при нескольких одновременных.
   - Добавить простой счётчик активных задач (в памяти или Redis).
   - Ограничить: максимум 2–3 одновременные генерации на весь сервер (или по пользователю).

4. **Бэкапы БД**
   - Нет автоматических бэкапов в скриптах деплоя.
   - Минимально: ежедневный скрипт `backup-db.sh` (sqlite3 .backup + rsync или копия в ~/backups).
   - Запуск через cron.

5. **Базовый мониторинг и алерты**
   - Нет уведомлений владельцу при падениях.
   - Минимум: 
     - Простой скрипт, который проверяет /api/health и шлёт в Telegram при ошибке.
     - Или использовать внешний монитор (UptimeRobot) + алерт.

## Приоритет 2 — Важно, но можно после первых тестеров

- Улучшить обработку ошибок и recovery зависших задач (уже частично есть скрипты).
- Проверить и усилить валидацию .env при старте (чтобы не падало молча).
- Мобильный UX (скролл при переходе в пакеты и т.д.).
- Обложки в фонотеке (если всё ещё проблема).
- Простая документация для тестеров (что можно/нельзя).

## Что НЕ критично для беты (отложить)

- Полная переделка фронтенда (монолит index.html).
- Глубокий Sentry / продвинутый мониторинг.
- Полноценный rate limiting на все эндпоинты.
- Бэкапы с ротацией и шифрованием.
- Rate limit + concurrent на уровне Redis (можно позже).

## Следующие шаги (предложение)

1. Завершить rate limiting (добавить серверную часть).
2. Закрыть guest trial bypass + параллельные генерации.
3. Добавить простой бэкап + базовый алерт.
4. Провести быстрый smoke-тест всего цикла (пробная → текст → песня → оплата → возврат).
5. Запускать бету.
